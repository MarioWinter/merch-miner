# Feature specs

This folder holds **mini-specs** for follow-up work that's too small for a
full PROJ-X workflow (which would otherwise mean `/requirements` +
`/architecture` skills, an entry in `features/INDEX.md`, and a 600-line
companion task file) but too large to live only in a PR description.

For full features (a new app surface, a new agent, a workflow with 50+
acceptance criteria) keep using the canonical pair under the repo root:

- `features/PROJ-X-name.md` — the spec, generated by `/requirements`
- `docs/tasks/PROJ-X-tasks.md` — the task list, generated by `/architecture`
- entry in `features/INDEX.md`

This folder is **only** for follow-up work attached to an existing PROJ.

## Structure

One sub-folder per parent PROJ:

```
docs/features/
  README.md                                ← this file
  proj-29/                                 ← Phase 1J + later follow-ups for PROJ-29
    drawer-side-by-side-mode.md
    topbar-niche-selector-and-floating-chat-removal.md
  proj-7/                                  ← future, when PROJ-7 grows a follow-up
    …
```

The sub-folder name matches the parent PROJ-X id (lower-cased,
hyphenated). The spec filename is short, kebab-cased, descriptive of the
scope — no `PROJ-` prefix needed (the sub-folder already encodes it).

## When to write one

Use this cutoff to decide where a piece of work lives:

| Scope | Where it lives |
|---|---|
| ≤ 2 acceptance criteria AND ≤ 2 files touched | No spec file — the PR description IS the spec. |
| 3–10 ACs OR 3+ files touched, single concern | A mini-spec in `docs/features/proj-X/<name>.md`. |
| > 10 ACs OR multiple concerns OR a new app surface | A full PROJ via `/requirements` + `/architecture` skills (NOT this folder). |

## Mini-spec template

Every mini-spec follows the same skeleton — see
`proj-29/drawer-side-by-side-mode.md` for a worked example.

```markdown
# <Concise title>

<Parent PROJ + 1-sentence context.>

## Why
<Why this change. Motivation, not implementation.>

## Current state — verified <YYYY-MM-DD>
<A table that pins down the as-is. File:line refs where it matters.>

## Acceptance criteria
- [ ] AC-1. <Atomic, testable statement.>
- [ ] AC-2. …

## Implementation steps
<Numbered list. One paragraph per step max.>

## Files to touch
<`Touch / Add / Delete` table or list.>

## Non-goals
<Bullet list of "explicitly out of scope".>

## Open questions
<Numbered list with proposed defaults. Used to gate user sign-off
before code is written.>

## Risk + rollback
<One paragraph each. Optional for low-risk diffs.>

## Status
- YYYY-MM-DD — <Spec written | Implementation queued | Shipped in PR #N>.
```

## Checkbox hygiene

When work ships, flip the `- [ ]` ACs to `- [x]` in the same PR that
delivers the code. The `## Status` line at the bottom is updated with
the PR reference. Leaves the spec as a self-contained record of WHAT
shipped and WHEN.
